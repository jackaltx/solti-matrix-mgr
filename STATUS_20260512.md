## Context

This collection grew organically during early Matrix homeserver exploration with no
upfront design. It works well in production but carries dead code, abandoned experiments,
and documentation that no longer reflects actual usage. This document captures what's
real vs. what's legacy so a future developer (or future-me) can make informed cleanup
decisions without having to re-derive the analysis.

Matrix has proven genuinely useful as a notification and capture platform. The collection
is worth maintaining — it just needs a focused cleanup pass.

## What's Worth Building Next (future ideas)

`matrix_config` is the most valuable module — declarative, idempotent, covers
the common case. Everything interesting builds on top of it. Some directions:

- **`matrix_config` token rotation** — currently tokens are written once on user
  creation and not rotated. A `rotate_token: true` flag on a user definition would
  invalidate old devices and generate a fresh token, useful for compromised bot accounts.

- **Room retention management** — `matrix_config` sets retention at room creation
  but cannot update it on existing rooms. An update path would let you tighten/loosen
  retention policy without recreating rooms.

- **`synapse_device_info` action module** — currently read-only (audit). A companion
  action module to revoke specific devices/tokens would complete the audit → remediate
  loop that `audit-tokens.yml` starts but cannot finish.

- **Conduit support** — the collection is Synapse-specific (Admin API paths, user
  format assumptions). The `matrix_config` module could grow a `server_type` parameter
  to target Conduit or Dendrite homeservers.

The collection works. `matrix_config` + `matrix_event` cover 90% of actual use.
The dead code is not causing harm — cleanup is cosmetic and can wait indefinitely.
When returning to this, Option B (archive) is the safest path: nothing is deleted,
intent is documented, and the active surface area becomes obvious.
